## Executive Summary

The Embr codebase shows **7 Major issues and 13 Minor issues** typical of growing social/marketplace platforms:

- **Database:** Missing indexes on high-cardinality foreign keys (participant1Id, participant2Id, followerId, followingId, authorId, senderId, postId). N+1 queries in follows/messaging services.
- **Frontend:** Unnecessary re-fetches due to unstable hook dependencies. No image optimization. No measurement of bundle size.
- **Cache:** Unbounded in-memory fallback could leak memory. No circuit breaker for Redis failures.
- **Cost:** Stripe, Mux, AWS SES not metered. Potential for bill shock without visibility.
- **Observability:** No database profiling, bundle analysis, or APM infrastructure.

**None are blockers today, but will become critical at 10k+ users.**

#### 1. Add Database Indexes on Foreign Keys
**Finding IDs:** f-perf-002, f-perf-015  
**Effort:** Low  
**Impact:** High

Missing indexes on frequently-queried columns:
- `Conversation.participant1Id`, `Conversation.participant2Id`
- `Follow.followerId`, `Follow.followingId`
- `Post.authorId`, `Message.senderId`, `Message.conversationId`, `Comment.postId`

**Query pattern:**
```sql
SELECT * FROM conversations WHERE participant1Id = ? OR participant2Id = ?  -- O(n) scan
SELECT * FROM follows WHERE followerId = ? AND followingId = ?             -- O(n) scan
SELECT * FROM posts WHERE authorId = ? ORDER BY createdAt DESC LIMIT 20   -- O(n) scan
```

**Recommendation:**
Add indexes in `apps/api/prisma/schema.prisma`:
```prisma
model Conversation {
  // ... existing fields
  @@index([participant1Id])
  @@index([participant2Id])
}

model Follow {
  // ... existing fields
  @@unique([followerId, followingId])  // Also prevents duplicates
}

model Post {
  // ... existing fields
  @@index([authorId])
}

model Message {
  // ... existing fields
  @@index([senderId])
  @@index([conversationId])
  @@index([senderId, createdAt])  // For ordering by date
}

model Comment {
  // ... existing fields
  @@index([postId])
}
```

Then run:
```bash
npx prisma migrate dev --name add_missing_indexes
npx prisma db push
```

#### 3. Fix Denormalized Count Atomicity
**Finding ID:** f-perf-013  
**Effort:** Medium  
**Impact:** Medium

**Issue:** follower/following counts are updated with separate queries; if one fails, counts desync.

**Current:**
```typescript
await Promise.all([
  this.prisma.profile.update({ where: { userId: targetUserId }, data: { followingCount: { increment: 1 } } }),
  this.prisma.profile.update({ where: { userId: userId }, data: { followerCount: { increment: 1 } } }),
]);
// If second fails, first is still applied → counts are out of sync
```

**Recommended (Prisma transaction):**
```typescript
await this.prisma.$transaction([
  this.prisma.profile.update({ where: { userId: targetUserId }, data: { followingCount: { increment: 1 } } }),
  this.prisma.profile.update({ where: { userId: userId }, data: { followerCount: { increment: 1 } } }),
]);
// Both succeed or both rollback
```

**Or (database trigger):**
```sql
CREATE TRIGGER follow_count_trigger AFTER INSERT ON follows
BEGIN
  UPDATE profiles SET followingCount = followingCount + 1 WHERE userId = NEW.followerId;
  UPDATE profiles SET followerCount = followerCount + 1 WHERE userId = NEW.followingId;
END;
```

#### 5. Implement Proper Cache Eviction
**Finding IDs:** f-perf-005, f-perf-006  
**Effort:** Low  
**Impact:** Medium

**Issue:** In-memory cache falls back with no size limit or scheduled cleanup.

**Current (cache.service.ts):**
```typescript
class InMemoryCache {
  private cache = new Map();
  cleanup() { /* exists but never called */ }
}
```

**Recommended:**
```typescript
class InMemoryCache {
  private cache = new Map();
  private readonly maxSize = 10000;
  private cleanupInterval: NodeJS.Timer;

  constructor() {
    // Schedule cleanup every 5 minutes
    this.cleanupInterval = setInterval(() => this.cleanup(), 300000);
  }

  set(key: string, value: any, ttl: number) {
    if (this.cache.size >= this.maxSize) {
      // Remove oldest entry (FIFO)
      const firstKey = this.cache.keys().next().value;
      this.cache.delete(firstKey);
    }
    this.cache.set(key, { value, expiresAt: Date.now() + ttl });
  }

  get(key: string) {
    const entry = this.cache.get(key);
    if (!entry) return null;
    if (entry.expiresAt < Date.now()) {
      this.cache.delete(key);
      return null;
    }
    return entry.value;
  }

  cleanup() {
    const now = Date.now();
    for (const [key, entry] of this.cache.entries()) {
      if (entry.expiresAt < now) {
        this.cache.delete(key);
      }
    }
  }

  onModuleDestroy() {
    clearInterval(this.cleanupInterval);
  }
}
```

**Redis Fallback Observability:**
```typescript
async get(key: string): Promise<any> {
  try {
    const value = await this.redis.get(key);
    return value ? JSON.parse(value) : null;

    this.logger.error(`Redis fallback for key ${key}`, error);
    this.metrics.increment('cache_redis_fallback_total');

  }
}
```

#### 7. Migrate Rate Limiting to Redis
**Finding ID:** f-perf-014  
**Effort:** Medium  
**Impact:** High

**Issue:** In-memory rate limiting ineffective in multi-instance deployments.

**Current (posts.service):**
```typescript
class RateLimitService {
  private rateLimits = new Map(); // In-memory; per-instance
}
```

**Recommended (Redis-backed):**
```typescript
async checkRateLimit(userId: string, action: string, limit: number, window: number): Promise<boolean> {
  const key = `rate:${action}:${userId}`;
  const current = await this.redis.incr(key);
  if (current === 1) {
    await this.redis.expire(key, window);
  }
  return current <= limit;
}

// Usage:
const allowed = await this.rateLimitService.checkRateLimit(userId, 'post_create', 10, 60);
if (!allowed) throw new TooManyRequestsException('Max 10 posts per minute');
```
