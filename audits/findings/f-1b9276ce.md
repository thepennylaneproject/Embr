## Title

useMessaging websocket ack promises can hang indefinitely

## Description

sendMessage and markAsRead return new Promise wrappers around socket acks but have no timeout or disconnect fallback. If server never acknowledges (disconnect/race), callers await forever and UI can remain blocked.

## Proof Hooks

### [code_ref] Promise without timeout in sendMessage

- Symbol: `sendMessage`

- Lines: 326-344

### [code_ref] Promise without timeout in markAsRead

- File: `apps/web/src/hooks/useMessaging.ts`

- Symbol: `markAsRead`

- Lines: 396-417


## Reproduction Steps

1. Establish websocket connection and trigger sendMessage.

2. Simulate network drop right after emit (or suppress ack handler server-side).

3. Observe returned promise never resolves/rejects.


## Impact

Pending UI operations can freeze and leave inconsistent loading states.


## Suggested Fix

**Approach:** Wrap ack promises with timeout rejection and socket disconnect/error listeners; optionally fall back to HTTP for critical operations.

**Affected files:** `apps/web/src/hooks/useMessaging.ts`

**Effort:** small

**Risk:** Medium; timeout value tuning affects UX.
