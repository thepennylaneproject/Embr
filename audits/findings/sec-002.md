# Finding: sec-002

> **Status:** open | **Severity:** High | **Priority:** P1 | **Type:** security | **Confidence:** Evidence

## Title

SQL Injection in $queryRaw without parameterization

## Description

Two database query endpoints use Prisma's `$queryRaw` with string interpolation for user-controlled parameters (userId, limit). The code uses direct `${}` template literal interpolation which is not parameterized, creating a SQL injection vulnerability. An authenticated attacker can craft a userId parameter containing SQL injection payload to exfiltrate all user credentials from the database.

## Reproduction Steps

1. Authenticate as any user (obtain valid JWT)
2. Call endpoint with SQL injection payload in userId parameter
3. Example: `GET /api/follows?userId=123' UNION SELECT ...`
4. Query returns data from injected SELECT statement
5. Attacker can enumerate users, passwords, email addresses

## Impact

**Severity: HIGH**

**Data Exposure:**
- All user credentials (emails, hashed passwords)
- User IDs and metadata
- Full database schema enumeration
- Potential access to other tables via UNION SELECT

**Attack Prerequisites:**
- Authenticated user (have valid JWT)
- Knowledge of database schema (easily guessable)

**Blast Radius:**
- Complete user database compromise
- Password hashes exposed (offline cracking)
- Email addresses for phishing campaigns
- User enumeration for targeted attacks

## Suggested Fix

**Approach:** Use Prisma's parameterized query mechanisms

**Option 1 (Recommended):** Rewrite using Prisma client
```typescript
// BEFORE (vulnerable)
const follows = await this.prisma.$queryRaw`
  SELECT f1.follower_id FROM follows f1
  WHERE f1.follower_id = ${userId}
`;

// AFTER (safe)
const follows = await this.prisma.follows.findMany({
  where: { followerId: userId }
});
```

**Option 2:** Use Prisma.sql with parameterization
```typescript
const follows = await this.prisma.$queryRaw`
  SELECT f1.follower_id FROM follows f1
  WHERE f1.follower_id = ${Prisma.sql([userId])}
`;
```

**Affected files:** 
- `apps/api/src/verticals/feeds/social-graph/services/follows.service.ts`
- `apps/api/src/verticals/feeds/social-graph/services/user-discovery.service.ts`

**Effort:** ~1 hour (replace both queries)

**Risk:** Low — Prisma client queries are safe by default

## Enhancement Notes

**Why This Happens:**
- Developers sometimes use `$queryRaw` for performance optimization
- Raw SQL can be faster than Prisma client for complex queries
- Mixing raw SQL with user input is dangerous without parameterization

**Prevention:**
- Always use Prisma client when possible
- If raw SQL needed, use `Prisma.sql` tagged template
- Code review checklist: Never use `${variable}` in SQL queries
- Linting rule: Flag `$queryRaw` usage without Prisma.sql

**Related Security Patterns:**
- Always parameterize database queries
- User input should never be concatenated into SQL
- Use ORM/query builder for 99% of cases
- If raw SQL required, use bind parameters or prepared statements
