# Finding: inv-004

> **Status:** open | **Severity:** Critical | **Priority:** P0 | **Type:** deployment | **Confidence:** Evidence

## Title

Deploy pipeline calls `nest build` which fails — API has never been continuously deployable

## Description

`.github/workflows/deploy.yml` line 25 runs:

```yaml
run: npm --prefix apps/api run build
```

`apps/api`'s `build` script executes `nest build`, which fails due to 666+ TypeScript strict-mode errors in the API codebase. This means:

1. **Every push to `main` triggers a deploy pipeline that cannot complete the build step.**
2. The API has never been deployable via the documented CI/CD pipeline.
3. In practice the API is deployed manually using `npx ts-node --transpile-only` — a pattern documented in `AGENTS.md` but not in the deploy workflow.

A technical reviewer who runs `git clone && npm run build` or examines the CI run history will immediately see a red pipeline. This is the most visible signal of accumulated technical debt for any investor.

Note: `f-deploy-build-failure-nestjs-001` was previously marked resolved, but that finding addressed the *API startup* failure (missing method, dev email validation). The *build pipeline* failure in `deploy.yml` remains open.

## Impact

**Severity: CRITICAL for credibility.**

- The API cannot be deployed via the documented pipeline.
- Any engineer cloning the repo who runs `npm run build` sees a failure.
- CI history shows a broken pipeline on every deploy trigger.

## Suggested Fix

**Approach:** Two options. Option B is recommended for the current phase.

**Option A — Fix all TypeScript errors** (multi-day, not recommended for investor sprint):
- Resolve all 666+ TypeScript strict-mode errors in `apps/api/src/`.
- `nest build` will then succeed natively.

**Option B — Officially adopt `--transpile-only` as the deploy strategy** (recommended, ~1–2 hours):
1. Update `apps/api/package.json` build script:
   ```json
   "build": "ts-node --transpile-only -r tsconfig-paths/register src/main.ts",
   "typecheck": "tsc --noEmit --pretty 2>&1 || true"
   ```
2. Update `.github/workflows/deploy.yml` line 25 to use `ts-node --transpile-only` instead of `nest build`.
3. Add a non-blocking `typecheck` step to `ci.yml` that reports TypeScript error count without failing the build.
4. Update `AGENTS.md` and `README.md` to document this as the intentional strategy (with a note that full TS migration is in progress).

**Affected files:** `apps/api/package.json`, `.github/workflows/deploy.yml`, `.github/workflows/ci.yml`  
**Effort:** 1–2 hours (Option B)  
**Risk:** Low — the API already runs this way in development. Option B formalizes the existing practice.
