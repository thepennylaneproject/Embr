# Finding: f-a50b7176

> **Status:** fixed_pending_verify | **Severity:** major | **Priority:** P1 | **Type:** bug | **Confidence:** evidence

## Title

Tool endpoints use one shared workspace for all authenticated users

## Description

Tool routes require authentication but `ToolService` resolved paths against a
single policy `workspace_root`. The current user id was not used to separate
sandboxes, so two distinct users on the same deployment could read and write
the same on-disk workspace.

## Proof Hooks

### [code_ref] Tool handlers ignore user identity beyond dependency presence

- File: `backend/app/api/routes/tools.py`
- Lines: 32 (original, pre-fix)

### [code_ref] Workspace resolved from policy only, no user_id segment

- File: `backend/app/tools/service.py`
- Lines: 181 (original, pre-fix)

## Reproduction Steps

1. Authenticate as user A and write a file via `POST /v1/tools/files`.
2. Authenticate as user B and read the same path via `GET /v1/tools/files/read?path=<same-path>`.
3. Before the fix, user B received user A's file content.
4. After the fix, user B gets HTTP 404 because their sandbox is distinct.

## Impact

Two distinct users on the same deployment can read and write each other's workspace files. This is a data-leakage vulnerability for any multi-user deployment.

## Suggested Fix

**Approach:** Derive `workspace_root` from `user_id` (or session) under a
base path. `WorkspacePolicy.workspace_root_for(user_id)` creates and returns
`<base>/<user_id>/` and validates that resolved paths cannot escape the
sandbox. `ToolService` and all route handlers now accept `user_id` as a
first-class parameter.

**Affected files:**
- `backend/app/tools/policy.py`
- `backend/app/tools/service.py`
- `backend/app/api/routes/tools.py`
- `backend/app/core/auth.py`

**Effort:** medium

**Risk:** Low ‚Äî behavioural change only for multi-tenant use; each user now
reads/writes their own isolated directory.
