## Title

useFeed permanently disables state updates after first effect cleanup

## Description

The cleanup sets isMountedRef.current = false, but the hook never resets it to true on subsequent effect runs. Because the effect depends on loadFeed (which changes as loading flags change), cleanup can run during normal lifecycle, making all later guarded state updates no-ops.

## Proof Hooks

### [code_ref] Cleanup flips mounted flag false

- File: `apps/web/src/hooks/useFeed.ts`

- Symbol: `useEffect cleanup`

- Lines: 260-266

### [code_ref] No corresponding reset to true before guarded updates

- File: `apps/web/src/hooks/useFeed.ts`

- Symbol: `isMountedRef usage`

- Lines: 56-94


## Reproduction Steps

1. Open a page using useFeed and trigger loadFeed once.

2. Cause effect re-run (e.g., parameter change or state path changing loadFeed dependencies).

3. Trigger refresh/loadMore and observe responses arrive but UI state remains stale due to guarded setState blocks.


## Impact

Feed can freeze and stop reflecting server data without crashing, creating persistent stale UI state.


## Suggested Fix

**Approach:** Reset isMountedRef.current = true at effect start (or remove this pattern and rely on AbortController/request-id guards). Also reduce effect dependency churn by stabilizing dependencies.

**Affected files:** `apps/web/src/hooks/useFeed.ts`

**Effort:** small

**Risk:** Low-medium; lifecycle handling changes can affect race behavior, so verify concurrent loads.


## Tests Needed

- [ ] Hook test: state updates still apply after effect re-run.

- [ ] Hook test: unmount still prevents setState warnings.

- [ ] Hook test: parameter changes do not freeze feed updates.
