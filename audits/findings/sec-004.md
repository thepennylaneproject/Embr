# Finding: sec-004

> **Status:** open | **Severity:** High | **Priority:** P1 | **Type:** security | **Confidence:** Evidence

## Title

Over-exposed user data in API responses leaks private fields

## Description

The `sanitizeUser` function in auth.service.ts only removes `passwordHash` and `googleId`, but returns user objects with all other fields including sensitive data. User profile responses include email addresses, full names, wallet balance summaries, and verification status that should not be exposed to all contexts.

An authenticated attacker can call `GET /api/users/:username` to retrieve any user's profile and build a complete enumeration of the platform's users with PII, emails, and financial data.

## Reproduction Steps

1. Authenticate as any user (obtain JWT)
2. Call: `GET /api/users/creator@embr.app`
3. Response includes:
   - email (PII)
   - fullName (PII)
   - isVerified (status info)
   - wallet object with balances (financial data)
   - follower/following counts (graph data)
4. Repeat for all users: enumerate entire platform

## Impact

**Data Leakage:**
- Email addresses exposed (enable phishing/targeting)
- Full names exposed (enable social engineering)
- Wallet balance summaries exposed (identify wealthy targets)
- Verification status exposed (identify status/reputation)
- Follower counts exposed (identify influencers)

**Attack Scenario:**
An attacker can:
1. Enumerate 1000s of users via `/users/:username` endpoint
2. Build spreadsheet: username, email, full name, wallet balance, follower count
3. Use for:
   - Targeted phishing of high-balance accounts
   - Data sale (email lists)
   - Account takeover attempts on weak passwords
   - Stalking/harassment of identified individuals

**Context-Based Exposure:**
- Public profile: Should show username, avatar, bio only
- Private profile: Should show more for self, less for followers
- Admin profile: Should show all including moderation flags

## Suggested Fix

**Approach:** Implement role-based response DTOs with context-aware filtering

**Affected files:**
- `apps/api/src/core/auth/auth.service.ts` (sanitizeUser function)
- `apps/api/src/core/users/users.service.ts` (user retrieval)
- Create new DTOs: `PublicUserProfileDto`, `PrivateProfileDto`, `AdminUserDto`

**Effort:** 2-3 hours (create DTOs + update 3-4 endpoints)

**Risk:** Low — defensive fix, no functional changes

## Tests Needed

- [ ] Public profile: Only username, avatar, bio are exposed
- [ ] Private profile: Additional fields (email, settings) visible to self only
- [ ] Admin profile: Moderation flags visible to admins only
- [ ] Endpoints return correct DTO based on requester context
- [ ] Wallet endpoint separate from profile (never expose balance in user profile)
- [ ] No email leakage in any context except to self

## Enhancement Notes

**DTO Pattern:**

```typescript
// What everyone can see
export class PublicUserProfileDto {
  id: string;
  username: string;
  avatar: string;
  bio: string;
  followerCount: number;
  followingCount: number;
  isVerified: boolean;
}

// What the user themselves can see
export class PrivateProfileDto extends PublicUserProfileDto {
  email: string;
  fullName: string;
  location?: string;
  birthDate?: Date;
  // User's own settings
  notificationPreferences?: any;
  privacySettings?: any;
}

// What admins can see
export class AdminUserDto extends PrivateProfileDto {
  role: string;
  suspiciousActivity: boolean;
  reportCount: number;
  banStatus?: string;
  createdAt: Date;
  lastLogin: Date;
}
```

**Usage Pattern:**

```typescript
async getProfile(
  @Param('username') username: string,
  @GetUser() requester: User,
) {
  const user = await this.usersService.findByUsername(username);
  
  // Determine which DTO to use
  if (requester.id === user.id) {
    return new PrivateProfileDto(user); // Self
  } else if (requester.role === 'ADMIN') {
    return new AdminUserDto(user); // Admin
  } else {
    return new PublicUserProfileDto(user); // Others
  }
}
```

**Wallet Data Separation:**

NEVER include wallet balance in user profile. Instead:
```typescript
// User profile endpoint
GET /api/users/:username → PublicUserProfileDto

// Separate wallet endpoint (requires auth)
GET /api/wallet → { balance, currency, ... }
```
