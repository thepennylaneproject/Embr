## Title

getMessages can build invalid createdAt filters on empty result windows

## Description

hasMoreBefore/hasMoreAfter use messages[messages.length-1]?.createdAt and messages[0]?.createdAt directly in Prisma where clauses. When messages is empty, undefined may be passed into lt/gt filters, risking runtime query validation errors.

## Proof Hooks

### [code_ref] Potential undefined dates used in Prisma count filters

- File: `apps/api/src/verticals/messaging/messaging/services/messaging.service.ts`

- Symbol: `getMessages hasMoreBefore/hasMoreAfter`

- Lines: 616-632


## Reproduction Steps

1. Request getMessages with before set to earliest message id or after set to latest id.

2. Force empty messages array result.

3. Observe whether Prisma throws validation error for undefined createdAt filter.


## Impact

Edge-page requests can fail unexpectedly instead of returning empty windows cleanly.


## Suggested Fix

**Approach:** Guard empty messages before issuing hasMoreBefore/After count queries; short-circuit to false when no anchor timestamp exists.

**Affected files:** `apps/api/src/verticals/messaging/messaging/services/messaging.service.ts`

**Effort:** small

**Risk:** Low.

- [ ] Integration test: no Prisma validation error on empty windows.
