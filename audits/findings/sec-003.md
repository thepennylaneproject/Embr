# Finding: sec-003

> **Status:** resolved | **Severity:** High | **Priority:** P1 | **Type:** security | **Confidence:** Evidence

## Title

Missing COOKIE_SECURE environment variable allows cookie transmission over HTTP

## Description

The authentication endpoints set JWT tokens as httpOnly cookies, but the secure flag was based on a missing or undefined `COOKIE_SECURE` environment variable. When undefined, cookies were set with `secure=false`, allowing them to be transmitted over unencrypted HTTP connections. An attacker on the same network could intercept these cookies and use them to impersonate users.

## Reproduction Steps

1. Start API without setting `COOKIE_SECURE` environment variable
2. Login or trigger OAuth callback: `POST /api/auth/login` or `GET /api/auth/google/callback`
3. Inspect response headers: `curl -i http://localhost:3003/v1/auth/login`
4. Notice `Set-Cookie` headers have `Secure=false` (or no Secure flag)
5. On a network with unencrypted HTTP, attacker can capture cookies

## Impact

**Security Impact:**
- JWT tokens transmitted over HTTP are vulnerable to interception
- Network attacker (WiFi, corporate network, ISP) can capture httpOnly cookies
- Tokens can be replayed to impersonate users
- Users' data, wallet, messages, and gigs become accessible

**Scope of Attack:**
- Login tokens: Access to all user data and actions
- Refresh tokens: Persistent access even after original token expires
- Risk multiplier: All 4 auth endpoints affected (login, OAuth, password change, email verification)

**Environment Specific:**
- Development: Often uses localhost HTTP (intentional)
- Production: Must always use HTTPS with secure cookies (critical)
- Staging: May use HTTP for testing, needs explicit configuration

## Implementation Details

### Root Cause

The original code pattern across 4 endpoints:

```typescript
res.cookie('accessToken', accessToken, {
  httpOnly: true,
  secure: process.env.COOKIE_SECURE === 'true',  // ← Undefined → false
  sameSite: 'strict',
  maxAge: ACCESS_TOKEN_COOKIE_MAX_AGE_MS,
});
```

**Problem:** When `COOKIE_SECURE` env var is not set, `process.env.COOKIE_SECURE === 'true'` evaluates to `false`, making cookies insecure by default.

### Solution Implemented

#### 1. Added Helper Method

```typescript
private getCookieOptions(maxAge: number) {
  const isSecure = process.env.NODE_ENV === 'production' 
    || process.env.COOKIE_SECURE === 'true';
  return {
    httpOnly: true,
    secure: isSecure,
    sameSite: 'strict' as const,
    maxAge,
  };
}
```

**Logic:**
- Production: Always secure (no need for env var)
- Development: `COOKIE_SECURE=true` to enable (for HTTPS testing)
- Fallback: Safe default (HTTP in dev, HTTPS in prod)

#### 2. Updated Validation Schema

```typescript
COOKIE_SECURE: Joi.boolean().default(false),
```

Allows `COOKIE_SECURE=true` to be set explicitly in `.env`.

#### 3. Refactored All 4 Endpoints

**Before:**
```typescript
res.cookie('accessToken', accessToken, {
  httpOnly: true,
  secure: process.env.COOKIE_SECURE === 'true',
  sameSite: 'strict',
  maxAge: ACCESS_TOKEN_COOKIE_MAX_AGE_MS,
});
```

**After:**
```typescript
res.cookie('accessToken', accessToken, 
  this.getCookieOptions(ACCESS_TOKEN_COOKIE_MAX_AGE_MS));
```

Endpoints updated:
1. `POST /auth/login` (line 49)
2. `GET /auth/google/callback` (line 85)
3. `PATCH /auth/change-password` (line 155)
4. `POST /auth/verify-email` (line 188)

### Environment Behavior

| Environment | NODE_ENV | COOKIE_SECURE | Result | Security |
|---|---|---|---|---|
| Production | `production` | (any) | `secure=true` | ✅ HTTPS only |
| Dev default | `development` | undefined/false | `secure=false` | ✅ Localhost HTTP OK |
| Dev HTTPS | `development` | `true` | `secure=true` | ✅ HTTPS testing |

### Verification

All 4 endpoints now consistently use:
- `httpOnly: true` — JavaScript cannot access tokens
- `secure: true` (in production) — HTTPS only transmission
- `sameSite: 'strict'` — No cross-site cookie sending
- Proper `maxAge` — Access token (15 min), refresh token (30 days)

### Defense in Depth

This fix combines with other security measures:

1. **Transport Layer:** Secure flag (this finding)
2. **HttpOnly Flag:** JavaScript access prevention
3. **SameSite Policy:** CSRF protection
4. **Short Expiration:** Limits token lifetime
5. **JWT Signature:** Validates token integrity

Even if one layer is compromised, others provide fallback protection.
