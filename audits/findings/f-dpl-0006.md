# Finding: f-dpl-0006

> **Status:** open | **Severity:** major | **Priority:** P0 | **Type:** bug | **Confidence:** evidence

## Title

Health checks are inconsistent with API routing and container runtime

## Description

API global prefix is set to /api, but Docker health checks probe /health; no dedicated health controller route was found in active API source. Additionally, compose healthcheck uses curl, while API runtime image does not install curl.

## Impact

Containers can report unhealthy despite app availability (or vice versa), causing restart loops and masking true production health.


## Suggested Fix

**Approach:** Add a real health endpoint under the configured prefix, standardize health probe path and tooling, and ensure runtime image has required probe binary (or pure node probe).

**Affected files:** `apps/api/src/main.ts` `apps/api/src//health.controller.ts (new)` `apps/api/Dockerfile` `docker/docker-compose.prod.yml`

**Effort:** small

**Risk:** Changing probe semantics can affect orchestrator restart behavior; validate in staging first.
