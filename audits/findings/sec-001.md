# Finding: sec-001

> **Status:** resolved | **Severity:** Critical | **Priority:** P0 | **Type:** security | **Confidence:** Evidence

## Title

Production secrets committed to git in .env file

## Description

The .env file at repository root contained real production credentials including database passwords, JWT secrets, Redis passwords, AWS keys, Stripe live keys, Mux tokens, and email credentials. This file was tracked in git history and accessible to anyone with repo access, creating a complete platform compromise risk.

## Reproduction Steps

1. Clone the embr repository
2. Examine `.env` file in repository root
3. Extract any `sk_live_*`, `sk_test_*`, database connection strings, or API tokens
4. These credentials grant access to:
   - PostgreSQL database (full read/write)
   - Redis cache (full read/write)
   - AWS S3 (upload/download files)
   - Stripe (process payments, access transactions)
   - Mux (process videos, access media)
   - Email system (send emails as embr)

## Impact

**Attack Vector:** Repository access → complete platform compromise

An attacker with read access to git history (employee departure, leaked credentials, supply chain attack) can:
- Access all user data (profiles, wallets, private messages, transaction history)
- Process fraudulent payments through Stripe
- Exfiltrate video content from Mux
- Send phishing emails impersonating embr
- Access AWS S3 bucket with media uploads
- Cache poison through Redis manipulation
- Modify database records directly

**Severity Justification:** Complete compromise of all external services and data; no defense once credentials are leaked.

## Suggested Fix

**Approach:** Multi-step secrets rotation and prevention

**Affected files:** `.env`, `.env.example`, `.pre-commit-config.yaml`, `docs/SECURITY.md`

**Effort:** 2-3 hours (rotation may take additional time if coordinating with team)

**Risk:** Low — only introduces new security best practices; no functional changes

## Resolution Details

### Actions Completed

1. **✅ Created `.env.example`** (4,959 bytes)
   - Comprehensive template with all 40+ environment variables documented
   - No real secrets, only field descriptions and setup instructions
   - Clear guidance on where to obtain each credential type

2. **✅ Implemented `.pre-commit-config.yaml`**
   - gitleaks integration for secret detection
   - Custom checks for common secret patterns (sk_live_, sk_test_, AKIA, etc.)
   - Prevents .env commit while allowing .env.example

3. **✅ Created `docs/SECURITY.md`**
   - Complete secrets management procedures
   - Step-by-step rotation guide for all external services
   - Incident response procedures
   - CI/CD best practices for secrets injection

4. **✅ Updated `.env` with placeholders**
   - All production credentials replaced with safe development values
   - Email validation issue fixed (config accepts dev emails)
   - Clear comments marking development-only values

### Manual Actions Required

**IMMEDIATELY (same day):**
1. Rotate database password in AWS RDS
2. Rotate Redis password in AWS ElastiCache
3. Regenerate JWT_SECRET and JWT_REFRESH_SECRET
4. Create new AWS IAM access keys, disable old ones
5. Revoke Stripe live keys, generate new ones
6. Revoke Mux API tokens, generate new ones
7. Revoke Google OAuth credentials, create new ones
8. Change SMTP email app password

**After rotation:**
1. Store all new credentials in AWS Secrets Manager or GitHub Secrets
2. Update CI/CD pipeline to inject credentials at deploy time (never in image)
3. Force-push commits if .env was ever committed (requires git-filter-repo on all team members)
